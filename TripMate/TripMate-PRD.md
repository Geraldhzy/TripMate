## 一、产品定位

旅行最美好的部分，从来不是做攻略、对表格、查路线——而是那些走着走着就发现的小巷、吃到第三碗还没饱的街边摊、在某个不知名的观景台突然感觉整个世界都对了的瞬间。

但在这之前，大多数人要花掉几倍于旅行本身的时间去准备：刷小红书收藏几百条、在表格里填来填去、出发前夜还在查第二天的路怎么走。旅行结束之后，照片在手机里积灰，想记录却不知道从哪里写起。

**TripMate 想做的，是让这些消耗本不该发生。**

> 一个真正懂旅行的 AI 伴侣——出发前帮你把行程想清楚，路上陪着你随机应变，回来后把那些零散的记忆拼成值得珍藏的故事。

TripMate 不是查询工具，也不是日程软件。它更像一个你信任的旅伴：你不需要问它「现在该去哪」，因为它已经知道了；你不需要担心计划临时变了怎么办，因为它会帮你重新想好；你只需要专注于当下——真正地在那里。

**核心差异**

| | 传统攻略工具 | TripMate |
|--|------------|---------|
| 行前 | 用户自己整合信息 | AI 理解你的偏好，主动规划，路线合理可执行 |
| 行中 | 用户对着攻略执行 | AI 感知实际状态，主动提醒和调整，用户只需走 |
| 行后 | 用户自己回忆写作 | AI 整理素材框架，用户填入真实感受，内容有温度 |
| 社区 | 看别人的攻略 | 行程可直接复制进自己的规划，从看到用无缝衔接 |

## 二、目标用户

**主力用户：有出行欲望、缺规划耐心的年轻人**

25–35 岁，工作忙但不愿放弃旅行。攻略收藏夹里存了几百条，真正出发前还是不知道怎么整合。不排斥 AI 帮忙，但也不想要一个千篇一律的机器人行程——想要的是「感觉是为我量身定做的」那种。

旅行结束后有分享欲，但从来没有时间和精力把游记写完。照片一直在手机里，感受慢慢就淡了。

**次要用户：预算敏感、爱探索小众的学生群体**

更灵活、更愿意临时改计划、对小众地点的兴趣高于网红景点。内容创作意愿强，但需要低门槛的工具帮助把旅行记录沉淀下来。

## 三、核心功能模块

### 模块 1：行前规划

**用户目标**：用最少的输入，得到一份「感觉是为我量身定做」的行程。

**主流程**：
```
输入目的地 + 出行日期 + 人数 + 偏好标签
        ↓
（可选）导入参考内容：小红书攻略文字 / 大众点评店铺信息
        ↓
AI 综合用户偏好 + 参考内容，生成结构化行程（每日主题 + 景点 + 时间 + 推荐理由）
        ↓
用户可编辑（增删景点、调换顺序、修改时间）
        ↓
点击「确认，出发日期到了提醒我」→ 行程保存，出行日当天自动切换沉浸模式
```

**关键交互**：
- 偏好标签选择：美食向 / 文化向 / 网红打卡 / 户外自然 / 懒人慢游
- AI 生成行程后展示「规划思路」，让用户理解为什么这样安排
- 支持「我想加一个 xxx」自然语言修改行程
- 行程以地图 + 时间轴双视图呈现，互为补充

---

**行程地图视图（重点功能）**：

行程规划的核心质量问题不是「去哪里」，而是「怎么走」——景点选得再好，顺序乱了一样让人精疲力竭。地图视图是行程质量的可视化验证器，也是让用户建立信心的关键设计。

**双视图切换**：

| 视图 | 内容 | 适用场景 |
|------|------|---------|
| **时间轴视图** | 按时间顺序展示每日行程列表，含时间、时长、推荐理由 | 默认视图，适合浏览内容细节 |
| **地图视图** | 在地图上标注每日各地点，连线显示移动路径 | 验证路线合理性，直观感受空间布局 |

**地图视图核心要素**：

```
地图画布
  │
  ├─ 📍 起点标注（每天出发地：酒店 / 交通枢纽）
  │
  ├─ 景点编号标注（① ② ③ 按顺序）
  │     每个标注点悬浮展示：
  │       · 地点名称
  │       · 建议游览时长
  │       · 到下一个地点的预计交通时间 + 方式
  │
  ├─ 路线连线（景点间有向箭头，直观显示移动方向）
  │     连线上标注：交通时间（如 "🚇 12分钟"）
  │
  └─ 🏁 终点标注（每天结束地：酒店 / 交通枢纽）
```

**AI 路线优化逻辑**：

AI 生成行程时，不只是挑选景点，必须同时完成路线优化：

| 优化维度 | 说明 |
|---------|------|
| **顺路原则** | 同一天的景点尽量聚集在相近区域，避免反复横跨城市 |
| **时间预算** | 每日行程总时长 = Σ（景点游览时长）+ Σ（交通时间）+ 用餐时间，控制在合理范围内（一般 9:00–21:00） |
| **起终点约束** | 将用户设置的住宿地点 / 交通枢纽作为每天的出发点和收尾点，路线从这里开始、绕回这里结束 |
| **交通方式选择** | 根据地点间距离推荐步行 / 地铁 / 打车，并估算时间 |
| **可行性兜底** | 若某天景点过于分散（不可避免），在「规划思路」中提前说明，给用户知情权 |

**起终点设置**：

用户在行程规划时可设置每日的出发 / 落脚地：
- **住宿地点**：输入酒店名称或地址，AI 以此为起终点规划每天路线
- **交通枢纽**：对于第一天（刚抵达）和最后一天（准备离开），起点/终点自动设为机场或火车站
- **不设置时**：AI 根据景点聚集区域自动推算合理的起终点区域（不强制要求用户填写）
- 已生成的行程本地缓存，**离线可查看**（路线优化、AI 对话等功能需联网）

**参考内容导入（重点功能）**：

用户在小红书、大众点评等平台收集到的攻略，往往散落在收藏夹和浏览记录里，规划时需要反复翻阅、手动整合——这是旅行准备中最耗时的环节之一。

本功能让用户将已有的参考内容「喂给」AI，由 AI 自动提取有效信息，融入行程规划。

**导入方式（v1 支持）**：

| 方式 | 操作路径 | 适用场景 |
|------|---------|---------|
| **iOS 系统分享** | 在小红书/大众点评 App 内点「分享」→ 选择 TripMate | 最自然，用户无需离开原平台 |
| **粘贴链接** | 在 TripMate 内粘贴复制的 URL | 分享后在应用内补充链接 |
| **粘贴文字** | 直接复制图文内容粘贴 | 兜底方案，覆盖所有平台 |

**技术处理策略（分层降级）**：

两个主流平台特性不同，采用分层策略确保 v1 可落地：

| 层级 | 触发条件 | 处理方式 |
|------|---------|---------|
| **抓取层** | 大众点评、公开可访问页面 | 服务端直接请求，提取店名、地址、评分、营业时间等结构化信息 |
| **解析层** | 小红书等 JS 渲染 / 需登录的页面 | 解析 URL 结构和页面 meta 信息（标题/描述），结合 AI 对热门景点餐厅的既有知识补全细节 |
| **兜底层** | 完全无法获取内容 | 提示用户手动复制关键段落，或仅保留链接来源标注 |

> 小红书限制说明：小红书内容页需登录且依赖 JS 渲染，服务端无法直接抓取正文。v1 通过「URL 元信息 + LLM 知识库」完成合理推断；后续版本探索 iOS Share Extension 在分享时传递富文本内容的可能性。

**AI 处理逻辑**：
- 从抓取/解析结果中提取：景点名称、餐厅/店铺、推荐理由、营业时间、地址等要素
- 将提取出的地点与用户指定的目的地、出行天数做匹配，合理插入行程
- 若参考内容与用户偏好有冲突（如用户选了「懒人慢游」但攻略景点密度很高），AI 在「规划思路」中说明取舍逻辑
- 无法识别的内容静默忽略，不影响行程生成

**交互形式**：
- 规划表单底部增加「+ 添加参考攻略」可折叠区域，默认收起，不增加首次使用门槛
- 链接解析完成后，显示识别到的要素摘要（如「已识别 3 家餐厅、2 个景点」），让用户感知到内容被消化
- 解析失败时展示友好提示，引导用户切换为粘贴文字方式
- 支持添加多条参考内容（每条来源独立展示，可单独删除）

### 模块 2：旅行中陪伴

**用户目标**：打开 App 就进入状态，AI 像一个真正懂我的旅伴，随叫随到，不用我动脑。

---

**沉浸式陪伴状态**

旅行开始后，每次打开 App 自动进入「旅行中」主界面，不需要用户手动切换。

**沉浸模式触发机制**：
- **自动触发**：出行日期当天 06:00，App 自动切换到旅行中主界面，并推送通知「今天出发了！行程已准备好 🎒」
- **提前手动触发**：用户也可以在行程详情页点击「现在出发」立即切换（应对计划提前出发的情况）
- **旅行结束**：最后一天行程结束后，次日自动退出沉浸模式，提示用户整理内容

界面以当前位置和今日行程为中心，AI 始终在线、随时可对话。

主界面核心信息层：
```
顶部：今天 Day X · 目的地 · 当前天气
中部：地图（当前位置 + 今日剩余景点标注）
底部：[ 📷 拍照 ] [ 🎥 录像 ] [ 🎤 录音 ]  ···  [ AI 对话 ]
         快捷记录区（常驻）              AI 陪伴入口
```

底部常驻两类入口，左侧是用户主动记录，右侧是 AI 对话，各自独立、互不干扰。用户可以随时点击底部对话框向 AI 提问，也可以随时拿起快捷记录留下这一刻。

---

**AI 陪伴能力优先级**

| 优先级 | 能力 | 触发方式 | v1 实现 |
|--------|------|---------|---------|
| 🔴 P0 | **景点到达感知 + 即时导览** | 用户手动打卡（v1）/ GPS 自动感知（v2） | ✅ v1 |
| 🔴 P0 | **快捷记录**（拍照 / 录像 / 录音） | 用户主动，一键触发 | ✅ v1 |
| 🔴 P0 | **下一步规划建议** | 完成打卡后、或用户主动询问 | ✅ v1 |
| 🔴 P0 | **偏差检测 + 重规划提醒** | 跳过景点 / 停留超时 | ✅ v1 |
| 🟠 P1 | **自由对话陪伴**（随叫随到） | 用户主动发起对话 | ✅ v1 |
| 🟠 P1 | **餐饮推荐**（临近饭点） | 时间触发 + 位置感知 | ✅ v1 |
| 🟠 P1 | **天气感知 + 行程预警** | 定时检查天气变化 | ✅ v1 |
| 🟡 P2 | **GPS 自动到达感知**（免手动打卡） | 进入景点地理围栏 | ⏳ v2 |
| 🟡 P2 | **停留时长智能提醒**（该走了） | 停留超过预期时长 | ⏳ v2 |
| ⚪ P3 | **周边实时发现**（展览/活动/限定） | 位置 + 时间触发 | ⏳ v3 |
| ⚪ P3 | **多人协作行程同步** | 组队旅行场景 | ⏳ v3 |

---

**P0：景点到达 + 即时导览**

用户打卡到达某景点时，AI 立即给出该景点的「陪伴卡片」：

- 拍照机位推荐（最出片的角度/时间）
- 1-2 个不踩雷的小贴士（排队策略、避开人流的时段）
- 当前天气相关提醒（如「今天光线很好，适合拍人像」）
- 周边 500m 内的餐厅/咖啡馆备选（用于后续用餐）

打卡时直接调用系统相机，拍摄的照片自动关联当前景点；完成打卡时触发震动反馈，增强仪式感。

---

**P0：快捷记录**

旅途中很多值得留下的瞬间不属于任何一个「景点」——路边偶然遇见的集市、夕阳突然打在某面墙上、坐在咖啡馆听到的一句话。如果记录的门槛稍微高一点，这些瞬间就消失了。

快捷记录的设计原则是**从打开到完成不超过 3 秒**，不需要选景点、不需要写文字，拿起来记就行。

**三种记录方式**：

| 方式 | 操作 | 自动附加信息 |
|------|------|------------|
| 📷 **拍照** | 点击直接开启相机，拍完即保存 | 时间戳 + 当前 GPS 位置 |
| 🎥 **录像** | 点击开启录像，再点停止，最长 3 分钟 | 时间戳 + 当前 GPS 位置 |
| 🎤 **录音** | 点击开始录音，再点停止，无时长上限 | 时间戳 + 当前 GPS 位置；自动转文字（后台异步，可选） |

**记录完成后**：
- 自动归入「今日日志」，按时间顺序排列
- 如果当前位置在行程景点范围内（500m），自动关联到该景点，方便后续内容创作时找素材
- 不在景点范围内的记录归为「自由记录」，同样保留在日志中
- 不弹任何确认框、不跳转页面，记完直接回到地图主界面

**与打卡的关系**：
- 快捷记录是**随手记**，不需要「到达某个景点」才能触发，完全由用户自主决定
- 打卡是**里程碑记录**，到了一个地方、想郑重标记一下时使用，会触发 AI 导览卡片
- 两者并行，互不替代

**P0：下一步规划建议**

完成打卡后，AI 主动展示「接下来去哪」的建议卡片：
- 推荐下一个景点（含预计出发时间、交通方式、途中耗时）
- 若当前时间已偏离原计划，给出调整后的路线
- 用户可一键接受建议，或开口说「我想去 xxx」

**P0：偏差检测 + 重规划**（详见模块 3）

**P1：自由对话陪伴**

底部常驻对话入口，用户随时可以问：
- 「附近有什么好吃的？」→ 结合位置 + 口味偏好推荐
- 「这个景点有什么历史背景？」→ 景点深度导览
- 「今天还能再加一个景点吗？」→ 实时评估时间余量后回答
- 「我有点累了，能调轻松一点吗？」→ 重新规划剩余行程节奏

AI 的回应风格：口语化、像朋友聊天，不是百科全书式堆砌。

**P1：餐饮推荐**

临近午饭 / 晚饭时间，且行程中未安排用餐，AI 主动推送：
- 基于当前位置的附近餐厅（结合用户偏好标签）
- 一键插入行程，自动调整后续景点时间

**P1：天气感知**

每日早晨 + 实时变化时主动推送：
- 「明天下午有雷阵雨，建议把 XX 室外景点调到上午」
- 「今天紫外线强，去 XX 记得做好防晒」

**旅行日志（v1）**

旅行日志是旅途中所有记录的汇聚地，按时间线自动整理，不需要用户主动「归档」。

每条日志来源：
- **快捷记录**：照片、视频、语音，记完即入流
- **打卡记录**：打卡时间 + 地点 + 用户标签 + 备注 + AI 补充的景点一句话描述

日志入口在主界面右上角，用户随时可以翻看今天走过的足迹。日志同时也是行后内容创作的素材库——发布游记或行程攻略时，所有记录都可以从这里调取。

### 模块 3：动态重规划

**用户目标**：计划赶不上变化，不想自己算，AI 帮我重新排好、只问我最关键的一个决定。

---

**触发方式**

重规划可由系统自动感知触发，也可由用户主动发起：

| 触发类型 | 具体场景 | 触发形式 |
|---------|---------|---------|
| **系统自动** | 完成打卡时检测到有景点被跳过 | Timeline 内嵌 Banner（非阻断） |
| **系统自动** | 当前停留时间明显超出计划时长 | Banner 提示「要不要调整后面的安排？」 |
| **系统自动** | 天气突变影响室外景点可行性 | 通知 + Banner 提示 |
| **用户主动** | 对话框中说「重新帮我规划一下」 | 直接进入重规划确认流程 |
| **用户主动** | 行程详情页点击「调整行程」按钮 | 直接进入重规划确认流程 |
| **用户主动** | 临时决定不去某个景点 | 长按景点卡片 → 「标记为不去了」→ 触发重规划 |

---

**重规划核心流程**

设计原则：**AI 做计算，用户做决策——但只做一次决策**。避免让用户逐一处理每个景点，而是先让 AI 给出推荐方案，用户只需要确认或微调。

```
触发重规划
        ↓
AI 自动整理当前行程状态（无需用户操作）：
  · 已完成：打卡记录中已完成的地点（自动识别，锁定不动）
  · 待确认：剩余未访问的景点，AI 预判哪些还来得及、哪些时间不够
        ↓
AI 生成推荐方案（含时间测算）：
  · 默认方案：保留最值得去的景点，删掉时间最紧张的，确保在计划时间内结束
  · 方案说明：一句话解释调整逻辑，如「删掉了 XX 因为过去要 40 分钟，时间来不及了」
        ↓
用户确认界面（三区分层，简化决策）
        ↓
用户点击「就这样，出发」→ AI 生成新行程
```

---

**用户确认界面设计**

界面分三个区域，信息清晰、操作极简：

```
┌─────────────────────────────────┐
│  ✅ 已完成（灰色，不可更改）       │
│     · 浅草寺  09:30 打卡          │
│     · 晴空塔  11:00 打卡          │
├─────────────────────────────────┤
│  ✨ AI 建议保留（默认勾选）        │
│     · 秋叶原         预计 13:30   │
│     · 东京塔         预计 17:00   │
├─────────────────────────────────┤
│  ⏭️ AI 建议跳过（默认不勾选）      │
│     · 银座    ← 去的话要多 1.5h   │
│     · 台场    ← 偏远，不顺路      │
└─────────────────────────────────┘
   预计结束时间：20:30  ✓ 在计划内

   [调整个别景点]    [就这样，出发 →]
```

**关键设计决策**：
- 已完成的景点锁定展示，给用户「成果感」，不会被重排打乱
- AI 预填推荐方案，用户只需要检查是否同意，默认路径是「一键确认」
- 「调整个别景点」展开后可以把「建议跳过」的景点手动拉回来，或把「建议保留」的景点移除——用户有完整控制权，但不强迫他们行使
- 底部实时显示预计结束时间，如果用户手动加回某个景点导致超时，结束时间变红色提示

---

**时间测算逻辑**

AI 在重规划时必须完成以下计算，确保方案可执行：

| 计算项 | 说明 |
|--------|------|
| **当前时间锚点** | 以用户当前实际位置和时间为起点，而非原计划时间 |
| **景点游览时长** | 使用原行程中设定的时长，用户可在编辑时调整 |
| **地点间交通耗时** | 调用地图 API 估算实时路况下的交通时间 |
| **用餐时间预留** | 若剩余行程跨越午饭/晚饭时段，自动预留 45–60 分钟 |
| **计划结束时间** | 以用户行程设定的每日结束地点（酒店/枢纽）为终点倒推 |
| **超时保护** | 若所有剩余景点都加入会超出计划时间，AI 优先删除最远/最耗时的景点，直到时间合理 |

---

**设计原则总结**

- **不打断当前体验**：Banner 提示不阻断用户正在进行的操作，用户可以选择稍后处理
- **AI 先给答案**：不让用户面对空白表单，而是先给一个可用的推荐方案
- **默认路径一步完成**：绝大多数情况下用户看一眼就能点「确认」，不需要逐个决策
- **用户保有控制权**：方案可以微调，但调整是「可选的」而不是「必须的」
- **时间透明可见**：整个确认过程中，预计结束时间始终可见，帮助用户自主判断取舍

### 模块 4：行后内容创作

**用户目标**：旅行中随手记录，结束后 AI 帮我把素材整理好，剩下的我来写。

**内容定位调整**：AI 的角色是「内容的骨架搭建者 + 素材整理者」，而不是代替用户写文章。每一段真实的文字、每一个情绪表达，都应该来自用户本人——这才是社区内容有价值的根本。

---

**两个创作入口**

旅行结束后，用户可以选择发布两种内容形态，形态不同，创作路径也不同：

| 内容类型 | 定位 | 核心形式 |
|---------|------|---------|
| **📝 图文游记** | 类小红书的传统图文内容 | 用户自己写的文章 + 照片，AI 提供框架和素材辅助 |
| **🗺️ 行程攻略** | 结构化的行程单 | AI 规划的行程为底，用户在每个景点/时段补充真实感受 |

---

**图文游记创作流程**

AI 的工作是把旅途中积累的素材整理好，搭起一个可以直接动笔的框架——而不是把文章写完。

```
旅行结束 → 进入「创作游记」
        ↓
AI 整理素材，生成内容框架：
  · 按天/按主题组织结构
  · 每个景点填入：打卡时间、用户选的标签、用户备注原文
  · 照片按 EXIF 时间自动归入对应景点
  · 空白处标注「在这里写写你的感受~」
        ↓
用户在框架上填写 / 修改 / 删减
  · 每个景点模块可独立编辑
  · 照片可拖拽调序、删除、补充
        ↓
发布 / 分享 / 导出
```

**AI 辅助边界**：
- ✅ AI 做：整理结构、归档素材、填入打卡记录、建议配图顺序、推荐 Hashtag
- ❌ AI 不做：替用户写正文段落、生成情感描述、补全用户没有记录的景点体验
- 用户没有留下备注的景点：框架中保留位置，但标注为空，引导用户回忆补充，而非 AI 杜撰

---

**行程攻略创作流程**

行程攻略是 TripMate 独有的内容形态：以 AI 规划的结构化行程为底，用户在每个节点上补充真实的一手信息，形成兼具「可复用性」和「真实温度」的攻略。

```
旅行结束 → 进入「发布行程攻略」
        ↓
展示完整行程单（AI 规划的骨架 + 打卡记录）
  · 已访问的景点：显示打卡时间 + 用户标签
  · 未访问的景点：显示「未到访」标记
        ↓
用户逐景点补充（每个景点独立卡片，选填）：
  · 实际用时（与 AI 预估对比）
  · 一句话点评（真实感受）
  · 避坑 / 亮点 Tips
  · 配图（从相册选取）
        ↓
确认发布
```

**行程攻略的信息诚信机制**：
- 未到访的景点在攻略中明确标注「未实际到访」，不展示用户补充入口
- 已访问景点的用户补充为「选填」，用户可以不写，但不会被 AI 内容填充
- 攻略详情页对读者展示：AI 规划 vs 实际执行的差异（如某景点实际花了 3 小时而非计划的 1 小时），帮助后来者做更准确的判断

### 模块 5：旅行内容社区

**用户目标**：看别人的旅行找灵感，分享自己的旅行被看见。

---

**内容双轨并行**

社区中同时流通两种内容，Feed 流中可筛选切换：

| 内容类型 | 浏览体验 | 核心价值 |
|---------|---------|---------|
| **图文游记** | 类小红书瀑布流，封面图 + 标题 + 标签 | 情绪共鸣、种草、找灵感 |
| **行程攻略** | 卡片式，目的地 + 天数 + 关键景点预览 | 可直接复用、参考规划 |

---

**Feed 流**

- 按目的地 / 旅行风格 / 内容类型（游记/攻略）浏览
- 游记卡片：封面图 + 标题 + 目的地 + 作者
- 攻略卡片：目的地 + 天数 + 景点缩略图 + 「X 人复制了这份行程」

---

**图文游记详情页**

- 用户自写的图文内容全文展示
- 评论 / 收藏 / 分享
- 若游记作者同时发布了对应行程攻略，底部显示关联攻略入口

---

**行程攻略详情页**

- 按天展示完整行程，每个景点显示：
  - AI 规划信息（时间、时长、推荐理由）
  - 作者实际打卡时间（如有）
  - 作者补充的点评 / Tips / 配图（如有）
  - AI 规划时长 vs 作者实际时长的对比标注
- **「复制这份行程」**：一键将攻略行程导入自己的规划，作为新行程的起点
- 评论 / 收藏 / 分享

---

**个人主页**

- 旅行足迹地图（去过的城市高亮）
- 已发布的游记 + 行程攻略列表
- 收藏的行程攻略

---

**社区冷启动策略**（v1 不做，标注供后续参考）：
- 邀请 KOL 旅行博主入驻，发布真实行程攻略
- 不用 AI 内容填充社区（与内容诚信原则一致）

| **周边实时发现** | 位置 + 时间触发 | ⏳ v3 | 限定活动、展览、快闪等实时信息 |

## 六、页面导航结构

### 整体导航框架

App 使用底部 Tab Bar 作为一级导航，共 4 个 Tab。Tab Bar 的可见性随场景变化：旅行中沉浸模式下 Tab Bar 收起，主界面全屏展示地图和 AI 对话。

**系统级入口（App 外）**：
- **Widget**：锁屏 / 桌面小组件，旅行中显示今日下一个景点 + 预计出发时间，一点直接进入沉浸主界面
- **系统通知**：重规划建议、天气预警、出发提醒等，点击直达对应功能页
- **Share Extension**：在小红书 / 大众点评 / Safari 中点「分享」可选 TripMate，将链接传入行程规划流程

```
┌─────────────────────────────────────────┐
│              App 顶层导航                 │
│                                          │
│  ┌──────────┬──────────┬──────────┬────┐ │
│  │  🗺️ 旅行  │ 🌍 发现  │ 📋 行程  │ 👤 我 │ │
│  └──────────┴──────────┴──────────┴────┘ │
└─────────────────────────────────────────┘
```

| Tab | 常态（无进行中行程） | 旅行中（沉浸模式） |
|-----|-------------------|-----------------|
| 🗺️ **旅行** | 创建新行程的规划入口 | 自动进入沉浸式陪伴主界面 |
| 🌍 **发现** | 社区 Feed，浏览游记/攻略 | 同左，供休息时刷攻略找灵感 |
| 📋 **行程** | 我的历史行程列表 | 我的历史行程 + 当前行程详情 |
| 👤 **我** | 个人主页、设置 | 同左 |
