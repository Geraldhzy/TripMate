1. PROJECT PURPOSE
================================================================================

TYPE: AI-Powered Conversational Travel Planning Application

WHAT IT DOES:
- Users chat with an AI agent to plan complete travel itineraries
- Real-time queries: flights, hotels, weather, exchange rates, visas, attractions
- Multi-agent architecture: main orchestrator + specialized flight/research agents

with real-time pricing, logistics, and practical information.

4. CORE DATA MODEL - TRIPBOOK
================================================================================

TripBook = Single Source of Truth (3-layer data model)

LAYER 1: DYNAMIC DATA (Cached Queries)
- flightQuotes: [{id, route, airline, price_usd, duration, stops, ...}]
- hotelQuotes: [{id, name, city, checkin, checkout, price_cny, rating, ...}]
- webSearches: [{query, summary, fetched_at}]

LAYER 2: USER CONSTRAINTS (What User Specified)
- destination: {value, cities[], confirmed, confirmed_at}
- departCity: {value, airports[], confirmed, confirmed_at}
- dates: {start, end, days, flexible, confirmed}
- people: {count, details, confirmed}
- budget: {value, per_person, currency, confirmed}
- preferences: {tags[], notes, confirmed}  ← pace (relaxed/balanced/packed)
- specialRequests: [{type, value, confirmed}]

LAYER 3: STRUCTURED ITINERARY (AI-Built Plan)
- phase: 0|1|2|3|4 (planning stage)
- theme: "海岛潜水·城市探索" (travel theme)
- route: ["Tokyo", "Kyoto", "Osaka"] (cities in order)
- days: [{day, date, city, title, segments[]}] (daily schedule)
- budgetSummary: {flights, hotels, meals, transport, activities, total_cny}
- reminders: ["Visa deadline: ...]
- practicalInfo: [{category, content, icon}]

PLANNING PHASES:
  Phase 1 (了解需求): Gather requirements, lock constraints
  Phase 2 (规划框架): Major transport, destination research (DELEGATE!)
  Phase 3 (完善详情): Fill POIs, restaurants, hotels
  Phase 4 (行程总结): Budget summary, final output

5. AI RESPONSE GENERATION - HOW IT WORKS
================================================================================

STREAMING PATTERN (NOT Request-Response):
All responses use SSE (Server-Sent Events) streaming, never batch responses.

AGENTIC LOOP (Main Agent - Max 30 Rounds):
┌─ Round 1
│  ├─ LLM call with system_prompt + messages
│  ├─ Receive: response_text + tool_calls (if any)
│  └─ If NO tool_calls → return response, END
│
├─ Round 2 (if tool_calls from Round 1)
│  ├─ Execute each tool_call
│  ├─ Sync results to TripBook
│  ├─ Add tool results to conversation as 'tool' messages
│  ├─ LLM call again with updated messages
│  └─ Receive: response_text + tool_calls
│
└─ Repeat until: LLM generates NO tool_calls → END

LLM STREAMING (OpenAI SDK):
```javascript
const stream = await client.chat.completions.create({stream: true, ...});
for await (const chunk of stream) {
  // Extract: chunk.choices[0].delta.content (text)
  //          chunk.choices[0].delta.tool_calls (partial functions)
  sendSSE('token', {text: content});
  accumulate tool_calls
}
```

SSE EVENT TYPES SENT TO FRONTEND:
- 'thinking' ........... AI is thinking (show spinner)
- 'token' .............. Text token (stream content live)
- 'thinking_done' ...... Done thinking
- 'tool_start' ......... Tool execution starting
- 'tool_result' ........ Tool completed (✅ label)
- 'tripbook_update' .... TripBook state changed (flights/hotels/itinerary)
- 'agents_batch_start' . Sub-agent delegation started
- 'agents_batch_done' .. Sub-agent delegation completed
- 'agent_error' ........ Individual sub-agent failed
- 'quick_replies' ...... AI-suggested follow-up questions
- 'round_start' ........ New agentic loop round
- 'done' ............... Response complete
- 'error' .............. Error occurred

FRONTEND RENDERING (Real-Time):
- 'token' → fullText += data.text → renderMarkdown() → DOM update → scroll
- 'tool_result' → show ✅ tool_name + label (e.g., "✅ 找到 3 个航班")
- 'tripbook_update' → updateItinerary() → right-side panel refreshes
- 'quick_replies' → show clickable suggestion buttons

6. TOOL SYSTEM & EXECUTION
================================================================================

AVAILABLE TOOLS (6 total):
1. web_search ............. Search the web (mock implementation)
2. search_poi ............. Find points of interest (restaurants, attractions)
3. search_flights ......... Flight search (returns mock data)
4. search_hotels .......... Hotel search (returns mock data)
5. update_trip_info ....... Write to TripBook (critical for state sync)
6. delegate_to_agents ..... Parallel delegation to sub-agents

TOOL ACCESS CONTROL (Important!):
  Main Agent:
    ✓ web_search, search_poi, search_hotels, update_trip_info, delegate_to_agents
    ✗ search_flights (BLOCKED - must use delegate_to_agents)
  
  Flight Agent (sub-agent):
    ✓ search_flights, web_search
    ✗ others
  
  Research Agent (sub-agent):
    ✓ web_search only
    ✗ others

TOOL EXECUTION FLOW:
┌─ LLM outputs: tool_calls = [{id, function: {name, arguments}}]
├─ For each tool_call:
│  ├─ Send SSE 'tool_start' {id, name, arguments}
│  ├─ Execute tool with timeout protection (~30s)
│  ├─ Sync results to TripBook:
│  │   - search_flights → tripBook.addFlightQuote()
│  │   - search_hotels → tripBook.addHotelQuote()
│  │   - web_search → tripBook.addWebSearch()
│  │   - update_trip_info → tripBook.updateConstraints/Itinerary/Phase()
│  ├─ Send SSE 'tool_result' {id, name, resultLabel}
│  ├─ Send SSE 'tripbook_update' {full_tripbook_state}
│  └─ Add to messages: {role: 'tool', tool_call_id: id, content: result}
└─ Loop back to agentic loop (Round 2+)

7. MULTI-AGENT SYSTEM - DELEGATION
================================================================================

MAIN AGENT ORCHESTRATES DELEGATION:
Main Agent (30 rounds max) can call:
  delegate_to_agents({
    tasks: [
      {agent: 'flight', task: 'Search flights from Beijing to Tokyo on 2026-05-01, 2 passengers'},
      {agent: 'research', task: 'Research Japan visa policy, weather, attractions for May 2026'}
    ]
  })

PARALLEL EXECUTION:
  ├─ Promise.all([
  │  ├─ runSubAgent('flight', task) → LLM loop with flight tools
  │  └─ runSubAgent('research', task) → LLM loop with research tools
  │ ])
  └─ Merge results

SUB-AGENT CONFIGS:
  Flight Agent:
    • Tools: search_flights, web_search
    • Max rounds: 3
    • Max tokens: 4096
    • Purpose: Self-direct flight search + airline research
  
  Research Agent:
    • Tools: web_search only
    • Max rounds: 2
    • Max tokens: 8192
    • Purpose: Comprehensive destination research (visas, weather, food, etc.)

DELEGATION RESULT STRUCTURE:
```javascript
{
  results: [
    {agent: 'flight', status: 'success', data: '...flight search results...'},
    {agent: 'research', status: 'success', data: '...research results...'}
  ],
  coveredTopics: ['航班搜索', '航线调研', '签证政策', '天气气候', ...],
  _instruction: '以上主题已由子Agent完成调研，主Agent禁止再用 web_search 重复搜索'
}
```

COVERED TOPICS TRACKING:
- Prevents main agent from re-searching already-covered topics
- Keeps main agent context efficient
- Sub-agents mark what they've researched, main agent honors it

const reader = resp.body.getReader();

// Split by newlines and parse SSE format
for (const line of lines) {
  if (line.startsWith('event: ')) eventType = line.slice(7);
  if (line.startsWith('data: ') && eventType) {
    data = JSON.parse(line.slice(6));

14. KEY CONCEPTS TO UNDERSTAND
================================================================================

1. SSE STREAMING
   - All responses stream via Server-Sent Events
   - NOT request-response or long-polling
   - Real-time token delivery creates live text appearance

2. AGENTIC LOOP
   - LLM runs in loop, making tool calls until complete
   - Each loop: LLM call → check tool_calls → execute → add to history
   - Safety limit: 30 loops maximum

3. TOOL INTERCEPTION
   - search_flights forbidden for main agent
   - Forces delegation to specialized flight agent
   - Maintains separation of concerns

4. COVERED TOPICS
   - Sub-agents mark what they've researched
   - Main agent receives this list
   - Prevents duplicate web_search calls
   - Keeps context efficient

5. STATELESS BACKEND
   - No session persistence on server
   - TripBook passed between client and server
   - Enables scaling without shared state

6. DYNAMIC PROMPTS
   - System prompt regenerated per request
   - Includes current date/time, trip context
   - Adapts to conversation flow

7. MULTI-AGENT PARALLELISM
   - Flight + research agents run simultaneously
   - Promise.all executes in parallel
   - Results merged after both complete

8. CLIENT-SIDE SECRETS
   - API keys never touch server
   - Stored only in browser localStorage
   - Only used for LLM API calls

9. REALTIME UI UPDATES
   - Right panel updates live as AI discovers info
   - TripBook snapshot streamed via SSE
   - Frontend reactively renders changes

10. MARKDOWN RENDERING
    - AI responses rendered as markdown
    - Supports bold, italics, links, lists, code blocks
    - Updates incrementally as tokens arrive
