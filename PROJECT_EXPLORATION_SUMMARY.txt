PROJECT NAME:      AI Travel Planner (对话式旅游规划助手)
PROJECT TYPE:      Conversational AI Web Application
LANGUAGE:          English (code), Chinese (UI)
PURPOSE:           Help users plan international trips via multi-turn AI chat
TARGET USERS:      Chinese travelers planning outbound trips

KEY INSIGHT:
This is a sophisticated AGENT-BASED system where a Main Agent orchestrates
conversation and can delegate specialized tasks to Sub-Agents (Flight & Research)
running in parallel. Uses SSE streaming for real-time UI updates.

USER WORKFLOW:

1. User opens app, configures API key (stored in browser localStorage)
2. Selects LLM provider (OpenAI/Claude/DeepSeek)
3. Starts conversation: "Plan a 5-day trip to Bali, budget 10k CNY"
4. AI engages in multi-turn dialog:
   • Asks clarifying questions (destinations, dates, budget, preferences)
   • Searches real-time information (flights, hotels, attractions)
   • Updates right-side itinerary panel as information is gathered
   • Generates structured day-by-day itinerary with times & logistics
5. User can refine or adjust plan
6. Trip details saved to sessionStorage (survives page reload)

✓ Hotel search & availability
✓ Destination research (attractions, restaurants, visa info)

✓ Itinerary visualization (day-by-day breakdown)
✓ Multi-provider LLM support (OpenAI, Claude, DeepSeek)
✓ Streaming responses (real-time text rendering)
✓ Session persistence (localStorage + sessionStorage)

REQUEST/RESPONSE FLOW:

User Message (Browser)
    ↓ [SSE + API Key + Provider + Model + TripBook snapshot]
Express Server (server.js)
    ├─ Validate & Sanitize (middleware)
    ├─ Restore TripBook state (from client snapshot)
    ├─ Build dynamic system prompt
    ├─ Call LLM (OpenAI SDK)
    │  └─ Stream response token-by-token
    │  └─ Detect tool calls
    ├─ Execute tools if called
    │  └─ web_search, search_flights, search_hotels, etc.
    │  └─ Sync results to TripBook
    ├─ Loop back if tools returned results
    └─ Send SSE events in real-time
    ↓ [SSE Event Stream: 'token', 'tool_result', 'tripbook_update', 'done']
Browser (EventSource)
    ├─ Append tokens to chat bubble (real-time typing effect)
    ├─ Show tool execution labels
    ├─ Update itinerary panel
    └─ Save snapshot for next request


DATA PERSISTENCE (TripBook):

The "TripBook" is the single source of truth:

Layer 1: Dynamic Data (ephemeral)
  • Flight quotes {airline, price, duration, stops}
  • Hotel quotes {name, price, rating}
  • Web search cache (prevents duplicate queries)

Layer 2: User Constraints (persistent across messages)
  • Destination (e.g., "Bali")
  • Departure city (e.g., "Shanghai")
  • Travel dates (start/end)
  • Party size (1-10 people)
  • Budget (amount + currency)
  • Preferences (beach, food, hiking, etc.)

Layer 3: Structured Itinerary (persistent)
  • Planning phase (1-4)
  • Day-by-day breakdown
  • Route/stops
  • Theme (e.g., "beach escape")
  • Budget summary
  • Reminders & practical info

KEY: TripBook lives on CLIENT (sessionStorage), not server database
     = Stateless backend, no DB needed
     = Client sends snapshot with each request


STREAMING PATTERN (SSE):

All responses use Server-Sent Events (not polling):

Server sends events in real-time:
  event: thinking       → LLM is processing
  event: token          → Text chunk "我会帮..."
  event: tool_start     → Tool executing
  event: tool_result    → Tool done
  event: tripbook_update → Panel update
  event: done           → Response complete

Frontend receives EventSource stream:
  eventSource.addEventListener('token', (e) => {
    appendText(JSON.parse(e.data).text)  // Real-time typing
  })

Result: ChatGPT-like experience with visible tool execution

HOW AI RESPONSES ARE GENERATED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP-BY-STEP:

1. FRONTEND SENDS REQUEST
   POST /api/chat with:
     • messages: [{role: 'user', content: 'Plan a trip...'}]
     • provider: 'openai' (or 'anthropic', 'deepseek')
     • model: 'gpt-4o'
     • tripBookSnapshot: {...}  // Previous state
     • Headers: X-Api-Key, X-Base-Url (optional)

2. SERVER VALIDATES & RESTORES STATE
   • Validates request (headers, schema, rate limits)
   • Creates/restores TripBook from snapshot
   • Builds dynamic system prompt (time, role, methodology, TripBook state)

3. OPENS SSE CONNECTION
   • Sets headers for streaming
   • Sends 'thinking' event

4. CALLS LLM VIA OPENAI SDK
   • Creates OpenAI client (with dynamic baseURL for provider)
   • Sends: system prompt + conversation history + tools
   • Temperature: 0.7 (balanced creativity)
   • Max tokens: 8192

5. STREAMS RESPONSE IN CHUNKS
   • For each chunk received:
     • If content: accumulate text + emit 'token' SSE event
     • If tool_calls: accumulate tool call parameters
   • Send 'thinking_done' when complete

6. CHECKS FOR TOOL CALLS
   • Parse tool call JSON
   • If special DSML format (DeepSeek R1): convert to OpenAI format
   • If no tools: send final response & exit

7. EXECUTES TOOLS (IF CALLED)
   • For each tool call:
     • Emit 'tool_start' SSE
     • Execute (e.g., web_search, update_trip_info)
     • Sync result to TripBook
     • Emit 'tool_result' + 'tripbook_update' SSE
   • Create tool response messages

8. LOOP BACK (IF NEEDED)
   • Add tool messages to conversation history
   • Next iteration: LLM sees user msg + assistant response + tool results
   • Decide: more tools? or final response?
   • Safety: max 30 rounds prevent infinite loops

9. FINAL RESPONSE
   • Emit 'done' SSE
   • Close connection

RESPONSE TIMES:
  • Simple text: 3-5 seconds
  • With 1-2 tools: 8-15 seconds
  • With agent delegation (flight + research): 20-40 seconds

STREAMING VS NON-STREAMING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STREAMING (SSE) - USED HERE:
  ✓ Real-time text delivery (like ChatGPT)
  ✓ Better perceived latency
  ✓ User sees thinking indicator
  ✓ Tool execution visible
  ✓ Parallel agent updates

Non-Streaming (not used):
  • Would require buffering full response
  • Return entire result as JSON
  • Slower perceived latency
  • No real-time feedback

1. MULTI-PROVIDER ABSTRACTION
   • Single OpenAI SDK client for OpenAI, Claude, DeepSeek
   • Dynamic baseURL switching based on provider
   • DSML fallback parser for DeepSeek R1 (XML tool calls)

2. AGENT DELEGATION SYSTEM
   • Main Agent delegates to specialized Sub-Agents
   • Parallel execution (not sequential)
   • Results merged into next iteration
   • Prevents re-delegation of same agent type

3. TRIPBOOK (3-LAYER DATA MODEL)
   • Layer 1: Dynamic quotes (ephemeral)
   • Layer 2: User constraints (persistent)
   • Layer 3: Structured itinerary (persistent)

4. CLIENT-SIDE PERSISTENCE
   • TripBook lives on client (sessionStorage)
   • Server stateless (no database)
   • Snapshot sent with each request
   • Enables continuity without DB

5. SSE STREAMING
   • Real-time token delivery (token-by-token)
   • Visible tool execution
   • Parallel agent status updates
   • ChatGPT-like UX

6. QUICK REPLIES DETECTION
   • Regex parser on LLM output
   • Extracts numbered lists
   • Suggests next steps to user

7. SYSTEM PROMPT DYNAMICS
   • Prompt changes per request
   • Includes current time, TripBook state, conversation context
   • Optimizes LLM behavior based on planning phase
