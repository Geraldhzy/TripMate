messages[] Array Evolution:
─────────────────────────
Initial State:
  [ {role: "system", content: "..."}, {role: "user", content: "..."} ]

After Round 1 - LLM calls tools:
  [ ..., {role: "assistant", tool_calls: [...]}, {role: "tool", ...}, ... ]

After Round 2 - LLM calls more tools:
  [ ..., {role: "assistant", tool_calls: [...]}, {role: "tool", ...}, ... ]

After Round N - LLM no tool calls:
  [ ... ] → RETURN fullText

Note: GROWS WITH EACH ROUND. LLM sees full history. No clearing.

╔════════════════════════════════════════════════════════════════════════════╗
║                      CRITICAL DESIGN PATTERNS                             ║
╚════════════════════════════════════════════════════════════════════════════╝

1. IMPLICIT ROUND SIGNALING
   ──────────────────────
   No explicit "round_start" or "round_end" event.
   
   Client infers from:
   ├─ token events = LLM thinking/responding
   ├─ tool_start → tool_result = tool execution phase
   └─ Back to token = new LLM round after tool results
   
   Design benefit: Minimal SSE overhead, state from event flow


2. MESSAGES ARRAY AS STATE MACHINE
   ────────────────────────────────
   Single source of truth: messages[] array
   
   ├─ Grows with each round
   ├─ Never cleared
   ├─ Contains full conversation history
   ├─ LLM sees all context
   └─ No external state management needed


3. PROVIDER ABSTRACTION
   ────────────────────
   Same loop logic, different implementations:
   
   ├─ streamOpenAI() vs streamAnthropic()
   │  └─ Different internal format → standardized output
   ├─ Both return {fullText, toolCalls, rawAssistant}
   ├─ Tool results formatted differently (lines 607-616)
   │  ├─ Anthropic: single message with array
   │  └─ OpenAI: one message per result
   └─ Client never sees differences


4. SELECTIVE TRIPBOOK SYNC
   ──────────────────────
   Not every tool result gets a special SSE event.
   
   ├─ rate_cached → exchange rates (line 300)
   ├─ weather_cached → weather data (line 310)
   ├─ tripbook_update → trip info (line 373)
   └─ Others → tool_result only (client doesn't need cache)
   
   Design benefit: Clients can cache reusable data


5. TIMEOUT PROTECTION
   ──────────────────
   withTimeout() wrapper (line 436-442)
   
   ├─ LLM timeout: 120s (LLM_TIMEOUT_MS) line 444
   ├─ Tool timeout: 30s (TOOL_TIMEOUT_MS) line 445
   ├─ Streaming timeout: per-round 120s
   └─ Prevents hung requests


6. DELEGATION LIMITS
   ─────────────────
   Prevents infinite delegation loops
   
   ├─ delegationCount tracks delegate_to_agents calls (line 565)
   ├─ Max 2 per request (line 594)
   ├─ Exceed limit → return error message
   └─ Forces LLM to complete with existing info
