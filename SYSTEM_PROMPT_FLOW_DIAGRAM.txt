KEY ARCHITECTURAL DECISIONS
═══════════════════════════════════════════════════════════════════════════════════

1. CLIENT-SIDE STATE PERSISTENCE
   ✅ tripBookSnapshot persisted to localStorage
   ✅ Restored on each request to recover conversation context
   ✅ Allows recovery from accidental browser refresh

2. SERVER-SIDE GLOBAL CACHES
   ✅ knownRates[] & knownWeather[] shared across sessions (TTL-based)
   ✅ Prevents duplicate expensive API calls
   ✅ BUT: Not included in tripBook directly (to avoid cross-session pollution)

3. INCREMENTAL UPDATES
   ✅ update_trip_info uses delta pattern (only changed fields)
   ✅ TripBook shallow-merges changes to preserve existing data
   ✅ Segments for same day preserved if new day data has empty segments[]

4. CONFIRMED FLAG AS GUARD
   ✅ confirmed: true → System prompt tells AI "don't ask again" (✅)
   ✅ confirmed: false → System prompt shows as pending (❓)
   ✅ Defaults to true if omitted (conservative: assume AI confirmed)

5. DELEGATION ARCHITECTURE
   ✅ Only flight searches delegated (specialized sub-agent)
   ✅ Other searches handled directly (web_search, search_poi, etc.)
   ✅ Sub-agent results capped at 4,000 chars to avoid context explosion

6. KNOWLEDGE REUSE
   ✅ Destination knowledge cached per session
   ✅ Re-injected if same destination mentioned in later turns
   ✅ No cross-session knowledge persistence (privacy)
