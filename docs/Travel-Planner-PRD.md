# AI Travel Planner - 产品需求文档 (PRD)

> 版本: 2.0 | 最后更新: 2026-04-16

---

## 1. 产品概述

### 1.1 产品定位

AI Travel Planner 是一款**对话式 AI 旅行规划助手**，通过自然语言对话帮助用户从零到一完成出境游行程规划。产品以 Web 应用形式提供，核心体验为「聊天即规划」——用户只需描述旅行需求，AI 自动调用工具获取实时信息（机票、酒店、景点、餐厅），渐进式构建完整行程方案。

### 1.2 目标用户

| 用户群体 | 典型场景 | 核心痛点 |
|---------|---------|---------|
| 出境游新手 | 第一次规划海外自由行，不知从何下手 | 信息分散、流程不清、怕踩坑 |
| 时间有限的上班族 | 想省时间，不想逐个比价、查攻略 | 规划耗时、信息碎片化 |
| 深度旅行爱好者 | 需要特色活动（潜水、徒步）的专业建议 | 专业信息难获取、定制化需求高 |

### 1.3 核心价值主张

- **一站式规划**: 从需求收集到预算汇总，全流程在对话中完成
- **实时数据驱动**: 机票报价、酒店价格、景点信息均为实时查询，非静态推荐
- **渐进式引导**: 4 阶段状态机，避免一次性信息过载，逐步深入
- **多模型支持**: 支持 OpenAI / DeepSeek / Kimi / GLM / MiniMax 等多家 AI 提供商（通过 OpenAI 兼容接口），用户自由选择
- **主从 Agent 架构**: 主 Agent 统筹规划，子 Agent 并行执行机票搜索和目的地调研，大幅缩短等待时间

## 2. 功能需求

### 2.1 核心功能

#### F1: 对话式旅行规划

**描述**: 用户通过自然语言描述旅行需求，AI 以对话形式引导规划。

**详细需求**:
- 支持中文自然语言输入
- AI 遵循 4 阶段渐进式规划状态机（见 2.3）
- 对话支持 SSE 流式输出，实时显示 AI 回复
- 支持快捷起步模板（如"日本东京京都大阪7天"、"西欧巴黎罗马之旅"等）

**验收标准**:
- 用户输入旅行需求后，AI 在 3 秒内开始流式响应
- AI 能正确识别目的地、日期、人数、预算等关键信息
- 对话过程中按状态机推进规划阶段

#### F2: 实时工具调用

**描述**: AI 在对话过程中按需调用专业工具获取实时信息。

**主 Agent 可用工具**:

| 工具名称 | 功能 | 数据源 |
|---------|------|--------|
| `web_search` | 搜索景点详情、住宿生态、实用信息 | Brave Search API |
| `search_poi` | 景点/餐厅/酒店地点搜索 | Google Maps API |
| `search_hotels` | 酒店价格查询 | Hotels API (Python 脚本) |
| `update_trip_info` | 更新行程参考书 | 内部状态 |
| `delegate_to_agents` | 并行委派任务给子 Agent | 子 Agent 系统 |

**子 Agent 独占工具**:

| 工具名称 | 所属 Agent | 功能 |
|---------|-----------|------|
| `search_flights` | flight Agent | 机票报价查询（含航线生态调研） |

> 主 Agent 不可直接调用 `search_flights`，必须通过 `delegate_to_agents` 委派给 flight Agent。

**验收标准**:
- 工具调用过程在前端以动画形式展示（开始/完成状态）
- 工具结果以人类可读标签显示（如"找到 5 条结果"、"找到 3 家酒店"）
- 子 Agent 执行过程通过 SSE 事件实时反馈进度

#### F3: 主从 Agent 架构

**描述**: 采用主从 Agent 分工模式，主 Agent 统筹规划，子 Agent 并行执行专项任务。

| Agent | 职责 | 可用工具 | 最大轮次 |
|-------|------|---------|---------|
| 主 Agent | 统筹规划、用户交互、行程构建 | web_search, search_poi, search_hotels, update_trip_info, delegate_to_agents | 12 |
| flight Agent | 航线生态调研 + 机票搜索 | search_flights, web_search | 4 |
| research Agent | 目的地综合调研（签证、交通、天气、美食） | web_search | 2 |

**委派规则**:
- 每次对话最多 2 次 `delegate_to_agents` 调用
- flight + research 两个 Agent 并行执行，互不等待
- 子 Agent 返回结果包含 `coveredTopics` 字段，主 Agent 不得重复搜索已覆盖主题
- 主 Agent 不可直接调用 `search_flights`（系统自动拦截）

#### F4: 行程参考书 (TripBook)

**描述**: 作为一次规划会话的 Single Source of Truth，分 3 层管理行程数据。

| 层级 | 名称 | 内容 | 说明 |
|------|------|------|------|
| Layer 1 | 动态数据 | 机票/酒店报价、web 搜索记录 | 工具结果自动同步 |
| Layer 2 | 用户约束 | 目的地、日期、人数、预算、偏好 | 用户确认后锁定，变更记录历史 |
| Layer 3 | 结构化行程 | 路线、每日计划、预算汇总、行前准备 | AI 逐步构建，按天级合并 |

**验收标准**:
- 约束变更自动记录历史（field/from/to/reason）
- TripBook 数据实时注入系统提示，AI 不重复询问已确认信息
- 前端右侧面板实时展示 TripBook 状态
- 支持 clearLevel 回退机制：重大变更清空全部（full），中等变更仅清空详情（details）

#### F5: 智能快捷回复 (Quick Replies)

**描述**: AI 回复后自动检测编号列表，生成可点击的快捷按钮。

**检测规则**:
- AI 回复末尾 800 字符内的编号列表（2-4 项，每项 ≤ 40 字）自动转为可点击选项
- 含问号的列表项自动排除（避免将问题误识别为选项）

#### F6: 行程概览面板

**描述**: 页面右侧实时展示行程关键信息的结构化面板。

**展示内容**:
- 目的地、出发城市、日期、人数、预算
- 规划阶段进度（Phase 1-4）
- 路线（城市间连线）
- 已查询的机票/酒店报价
- 每日详细行程（segments: 景点/餐饮/交通/住宿/航班）
- 预算汇总、行前准备、实用信息

#### F7: 历史行程管理

**描述**: 对话历史保存在浏览器本地，支持多行程切换。

**需求**:
- 自动保存对话记录到 localStorage
- 左侧抽屉展示历史行程列表
- 支持切换、新建对话
- TripBook 快照随对话保存/恢复

#### F8: 多模型设置

**描述**: 用户可选择 AI 模型提供商和具体模型。

**支持的模型**:

| 提供商 | 默认模型 | 接入方式 |
|--------|---------|---------|
| OpenAI | GPT-4o | OpenAI SDK |
| DeepSeek | deepseek-chat | OpenAI 兼容 (自定义 Base URL) |
| Kimi (月之暗面) | moonshot-v1-auto | OpenAI 兼容 |
| GLM (智谱) | glm-4-plus | OpenAI 兼容 |
| MiniMax | MiniMax-Text-01 | OpenAI 兼容 |

**配置项**:
- API Key（本地存储，不上传服务器）
- 自定义 Base URL（支持代理/第三方兼容 API）
- Brave Search API Key（用于 web_search 工具）

### 2.2 渐进式规划状态机（4 阶段）

| 阶段 | 名称 | 目标 | AI 行为 | 完成标志 |
|------|------|------|--------|---------|
| Phase 1 | 了解需求 | 收集并确认用户核心需求 | 提问确认目的地、日期、人数、预算、偏好 | 关键约束基本明确 |
| Phase 2 | 规划框架 | 大交通方案 + 行程框架 | 委派 flight + research Agent 并行搜索，确定路线和每日主题 | 大交通 + 行程框架确定 |
| Phase 3 | 完善详情 | 细化每日行程安排 | 并行搜索景点/餐厅/酒店，一次性写入所有天的 segments | 所有天的详细行程已写入 |
| Phase 4 | 行程总结 | 预算汇总 + 行前准备 | 生成分类预算、行前清单、实用信息 | budgetSummary + reminders + practicalInfo 已写入 |

**阶段推进规则**:
- Phase 只通过 `update_trip_info(phase=X)` 变更，严禁跳阶段
- 每个阶段完成后必须等用户确认才推进
- 用户中途改需求时支持按需回退（重大变更回退到 Phase 1，中等变更回退到 Phase 2）

**Phase 3 并行搜索策略**:
- 每轮最多并行发起 **5 个**工具调用（web_search / search_poi / search_hotels 混合）
- 例如 7 天行程只需 2-3 轮搜索即可全部覆盖，而非逐天逐个搜索
- 搜索完成后一次性调用 `update_trip_info` 写入所有天的 segments

### 2.3 信息来源与可信度

| 信息类型 | 来源方式 | 可信度保障 |
|---------|---------|-----------|
| 签证政策 | research Agent → web_search | 标注来源链接 |
| 机票价格 | flight Agent → search_flights | 实时报价 |
| 酒店价格 | 主 Agent → search_hotels | 实时报价 |
| 景点信息 | 主 Agent → web_search + search_poi | 交叉验证，标注来源 |
| 天气/交通/美食 | research Agent → web_search | 子 Agent 并行调研 |

**价格展示规则**: 所有价格同时标注当地货币和人民币。

**来源标注规则**: 签证政策、门票/开放时间、价格等关键信息必须标注来源 `[来源: 显示文字](URL)`。

## 3. 非功能需求

### 3.1 性能

- 首字响应时间: SSE 流式输出，首 token < 3 秒
- 工具调用超时: 单次工具调用上限 30 秒
- Agent Loop 上限: 主 Agent 单次请求最多 12 轮工具调用
- 子 Agent 超时: 单个子 Agent 执行上限 120 秒
- LLM 调用超时: 单次 LLM API 调用上限 300 秒

### 3.2 安全

- API Key 仅存储在浏览器 localStorage，不持久化到服务器
- API Key 通过请求头 `X-API-Key` 传输（需 HTTPS 环境保障安全）
- 服务端不存储任何用户对话数据
- 请求输入通过 Joi Schema 严格校验和清洗
- Helmet + CORS + 自定义安全头保护
- 速率限制: 通用 100 次/小时，聊天 20 次/小时，工具 50 次/小时

### 3.3 可用性

- 响应式布局: 聊天面板 + 行程面板自适应
- 中文界面，所有 AI 回复使用中文
- 工具调用状态可视化，用户清晰感知进度
- 子 Agent 执行进度实时 SSE 推送

### 3.4 兼容性

- 现代浏览器（Chrome/Firefox/Safari/Edge 最新版）
- Node.js >= 18

## 4. 用户流程

```
用户打开应用
    |
设置 AI 模型 & API Key & Brave API Key
    |
输入旅行需求 / 点击快捷起步模板
    |
Phase 1: AI 提问确认约束 <-> 用户通过 Quick Reply 或自由输入回答
    |
Phase 2: AI 并行委派 flight + research Agent
         -> flight Agent 搜索机票
         -> research Agent 调研签证/交通/天气/美食
         -> AI 整合结果，展示大交通方案 + 行程框架
         -> 用户确认/调整
    |
Phase 3: AI 并行搜索景点/餐厅/酒店（每轮最多 5 个并行调用）
         -> 一次性写入所有天的详细 segments
         -> 右侧面板实时展示每日行程
         -> 用户确认/调整
    |
Phase 4: AI 生成预算汇总 + 行前准备 + 实用信息
         -> 用户确认，规划完成
```
